# CLAUDE.md — uaa-singular

## System Overview

`uaa-singular` is a small, dependency-light **client-side JavaScript library** ("Singular") that lets a web application detect and track the identity of a user authenticated against Cloud Foundry's **UAA** (User Account and Authentication) identity server. It implements the browser side of an **OIDC Session Management (check-session iframe) pattern**: a hidden "OP iframe" (served by UAA) and a hidden "RP iframe" (served by this library) exchange `postMessage` events to silently detect login/logout/session-changed events, then perform silent OAuth implicit-grant redirects to fetch fresh ID tokens / access tokens. There is no server-side or backend component — the entire application is event-driven, iframe-based browser code, packaged as a UMD/global bundle and consumed as a static asset by any web app.

## Core Tech Stack & Tools

- **Language:** Vanilla ES5/ES6 JavaScript (no TypeScript, no framework — plain `var`/`function` style, CommonJS `require`/`module.exports` mixed with ES `import`/`export default`)
- **Bundler:** Webpack 5 (`webpack-cli`) — produces 3 separate bundles from `webpack.config.js`
- **Polyfill:** `idempotent-babel-polyfill` (injected as a Webpack entry point, not via Babel transpilation config)
- **`jwt-simple`:** declared as a production dependency in `package.json`, but `src/` never imports it — the library hand-rolls JWT decoding in `singular.js` via `atob`. Its only actual consumer is the Nightwatch test helper `test/tests/accessTests.js`, which `require`s it directly to decode tokens for assertions.
- **Unit testing:** Jasmine 6 (`jasmine.json` config, specs in `test/spec/`)
- **Integration/E2E testing:** Nightwatch 3 + ChromeDriver + Selenium standalone server, driving a real UAA instance and a real Chrome browser headlessly
- **Local static server:** `http-server` (serves the built `singular/` bundle for integration tests)
- **CI:** GitHub Actions (`unit-test.yml` — build + unit test on every push) and CodeQL (`codeql-analysis.yml`)
- **Dependency management:** Dependabot (`.github/dependabot.yml`)

## Key Architecture & Data Flow

Singular runs entirely in the browser across **several cooperating iframes/windows**, coordinated by `window.postMessage` and silent OAuth2 implicit-grant redirects:

1. **Host page → `Singular.init(params)`** (`singular.js`): validates required config (`uaaLocation`, `clientId`) via `uaaValidator.js`, then creates two hidden iframes and appends them to `document.body`:
   - **OP iframe** → UAA's `/session_management` endpoint (server-managed OIDC session-check page)
   - **RP iframe** → this library's `client_frame.html`, which boots `rpFrame.js` and itself contains a nested, initially-blank `authorizeFrame` iframe used for silent redirects (step 3)
2. **RP iframe polls for session changes** (`rpFrame.startCheckingSession`): every `checkInterval` ms, it `postMessage`s the client ID + last known `session_state` to the OP iframe.
3. **OP iframe replies** with `changed` / `unchanged` / error via `postMessage`, handled by `rpFrame.receiveMessage`:
   - `changed` → clears cached identity, silently redirects a nested `authorizeFrame` to UAA's `/oauth/authorize` (implicit grant, `response_type=id_token`, `prompt=none`) targeting **`postauth.html`**.
   - `unchanged` (first check only) → hydrates identity from `localStorage`.
4. **`postauth.html`** loads after the silent redirect completes and calls `window.parent.afterAuthorize()`, which extracts the `id_token` + `session_state` from the URL fragment, decodes the JWT (`Singular.decodeJwt`), persists it to `localStorage`, and fires the host's `onIdentityChange` / `onLogout` callback.
5. **`Singular.access(scope)`** repeats the same silent-redirect pattern independently — `rpFrame.fetchAccessToken` opens a one-off iframe against `/oauth/authorize?response_type=token`, redirecting to **`postaccess.html`**, which calls `window.parent.afterAccess(index)` to resolve/reject the caller's Promise with the access token.

There is no server, no router, no database — "controller" and "service" layers are collapsed into `singular.js` (public API) and `rpFrame.js` (session/token engine), with `uaaValidator.js` acting as a guard-clause layer before any network calls are trusted.

## Development Setup & Crucial Commands

**Requirements:** Node.js + npm, Chrome (for headless integration tests via ChromeDriver), git (integration tests clone the real `cloudfoundry/uaa` repo, which additionally requires a JDK to build/run).

| Command | Purpose |
|---|---|
| `npm install` | Install dependencies |
| `npm run build` | Wipe and rebuild `singular/` via Webpack (all 3 bundles) |
| `npm run unit-test` | Run Jasmine unit specs (`test/spec/`) — no browser/UAA needed |
| `npm run get-uaa` | Clone `cloudfoundry/uaa` (`develop` branch) into `test/tmp/uaa` |
| `npm run start-uaa` | Build & boot a real UAA server in the background (`test/startUAA.sh`) |
| `npm run stop-uaa` | Kill the background UAA server |
| `npm run start-singular-app` | Serve the built `singular/` bundle via `http-server` in the background |
| `npm run stop-singular-app` | Kill the background static server |
| `npm run integration-test` | Run Nightwatch/ChromeDriver E2E specs against the running UAA + static server |
| `npm test` | `build` → `unit-test` → `integration-test` (full suite; requires UAA/http-server already running per README) |

Full integration run sequence (per README/DJM-README):
```bash
npm install
npm run get-uaa
npm run start-uaa
npm run start-singular-app
npm test
npm run stop-singular-app
npm run stop-uaa
```

CI (`unit-test.yml`) only runs `npm run build` + `npm run unit-test` on every push — it does **not** run the Nightwatch integration suite.

## Guardrails & Design Patterns

- **`singular/` is generated, not authored.** All real changes belong in `src/`; never hand-edit files under `singular/` — they're overwritten by `npm run build` and are what gets published to npm (`package.json` `files`/`main`).
- **Keep the public surface on `singular.js` minimal.** `singular.js` is the only file the host page talks to (`window.Singular`). Session/token mechanics belong in `rpFrame.js`, which runs isolated inside the RP iframe and is never referenced directly by host-page code.
- **Validate before trusting the network.** `uaaValidator.js` exists specifically to fail fast with a clear error when `uaaLocation` isn't a reachable, version-compatible UAA, or when the OAuth client's `redirect_uri` isn't whitelisted — preserve this guard-clause-first style rather than letting failures surface as opaque postMessage/XHR errors.
- **Dependency injection for testability.** Both `singular.js` (`getUaaValidator`/`setUaaValidator`) and `rpFrame.js` (`getUaaValidator`/`setUaaValidator`) expose swappable validator seams purely so Jasmine specs can stub network calls — follow this pattern rather than reaching for a mocking framework.
- **No secrets, all implicit grant.** This library intentionally never holds a client secret — it only uses OAuth2 implicit grant (`response_type=id_token` / `token`) with `prompt=none` silent redirects. Do not introduce flows that require a client secret in browser code.
- **State lives in `localStorage`, keyed by `storageKey`.** Identity claims and session state are cached under `${storageKey}-claims` / `${storageKey}-session_state`; treat these as the only persistence layer — there is no cookie or server session to fall back on.
- **Code style is intentionally old-school ES5/ES6-mixed** (`var`, `function` expressions, CommonJS `require` alongside ES `import`/`export default`) to keep the bundle small and broadly browser-compatible (see README's IE11+ support matrix) — don't introduce modern syntax (`class`, `async/await`, arrow functions in the library bundle) without checking the target browser matrix in the README.
- **Unit specs mirror `src/` 1:1** (`test/spec/rpFrameSpec.js`, `singularSpec.js`, `versionUtilSpec.js`); integration tests live separately under `test/tests/` and require a live UAA + built bundle — keep new logic testable at the unit level rather than pushing everything into Nightwatch.
- **`chromedriver`'s pinned version in `package.json` must support your locally installed Chrome's major version**, or `npm run integration-test` fails immediately for every test with `session not created: This version of ChromeDriver only supports Chrome version X`. Never "fix" this by committing a downgraded/pinned version to work around one contributor's browser — `chromedriver` is a shared dependency for every contributor and CI; pin changes belong in a real dependency-bump PR, not as a side effect of a local test run. Update your local Chrome instead, or override the version locally without committing.
- **UAA's local test harness no longer uses the Gradle Cargo plugin.** `test/startUAA.sh` boots it with `./gradlew run &` (UAA's own task, which kills any stale UAA process before running `bootRun`) and `stop-uaa` kills it via UAA's `scripts/kill_uaa.sh`. `bootRun` blocks in the foreground (unlike the old Cargo deploy task), so it must stay backgrounded — dropping the trailing `&` makes `start-uaa` hang forever before it ever reaches the readiness poll.
- **`test/tests/accessTests.js` can be flaky when run in the same Nightwatch invocation as `identityTests.js`** — a timing-sensitive OAuth silent-redirect can occasionally resolve with a null token/error, failing `jwt.decode` with `No token supplied`. Before treating this as a regression, re-run it in isolation: `cd test && ../node_modules/nightwatch/bin/nightwatch --test tests/accessTests.js`.
